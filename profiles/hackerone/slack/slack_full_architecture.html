How it all fits together: Slack started as a simple LAMP stack (Linux, Apache, MySQL, PHP) and evolved over a decade into a distributed architecture. The PHP monolith still handles most business logic, but is backed by Vitess-sharded MySQL, Kafka job queues, a Java real-time service, and a custom edge cache called Flannel — each solving a specific scaling bottleneck.

The web app was originally CoffeeScript with vanilla DOM APIs. It migrated to React + async/await ES6, and is now progressively moving to TypeScript for type safety. The state management model maintains a local snapshot of the entire workspace — all channels, members, and preferences — in memory (RAM) after the initial load.

Desktop app

Electron
Hybrid approach
Remote asset loading
Hybrid model: Unlike apps that bundle everything locally, Slack's Electron app ships some core assets but loads most code remotely. This means updates are instant — no reinstall needed — while startup-critical code is bundled for speed. The desktop app wraps the same React web codebase inside Electron's Chromium shell.

Mobile apps

Swift (iOS)

Modular architecture
In 2020, Slack undertook a major mobile refactor — Stabilize, Modularize, Modernize — breaking a large monolithic mobile codebase into loosely coupled modules. Mobile clients use the REST Web API for most actions (instead of WebSockets) to avoid the overhead of maintaining persistent connections on mobile networks.

Backend stack

Core application server

PHP 5 → HHVM → Hack
XHP (server-side HTML)
REST Web API
The PHP monolith is Slack's central brain — it handles all CRUD operations on the database (channels, messages, users), processes API calls, and manages authentication. It started as PHP 5, migrated to HHVM (which runs PHP much faster), and then moved to Hack — Facebook's PHP superset — for type safety, async support, and better tooling. All client interactions go through this monolith first.

Async job queue system

Apache Kafka
Redis (job queue)
Kafkagate (Go service)
JQRelay
Activities like unfurling links, indexing messages for search, and sending notifications are too slow to handle in real-time — they get pushed to an async job queue. Originally just Redis queues, Slack added Kafka in front for scale. A Go service called Kafkagate accepts job submissions via HTTP POST and enqueues them to Kafka. A relay service called JQRelay pulls jobs from Kafka and drops them into Redis, ensuring each topic has exactly one relay process.

Search

Apache Solr
Sharded by channel_id
Message search is powered by Apache Solr, partitioned using channel_id as the shard key — the same strategy used for the database — so message queries and search queries hit the same logical shard, keeping things consistent and avoiding cross-shard fan-out.

Database architecture

Evolution: LAMP → sharded MySQL → Vitess

1
Early days: single MySQL (LAMP)
The original database was a single MySQL instance — fast to set up, but immediately hit limits as workspaces grew into thousands of users.
2
Sharded MySQL (active-active)
Slack manually sharded MySQL by workspace — each workspace was assigned to a database shard and a message server shard. Two nodes per shard in different data centers replicated to each other asynchronously. When one failed, the other took over. This worked until shards themselves became hot spots.
3
Vitess (2017 → present)
Vitess is a sharding and topology management layer that sits on top of MySQL. Slack migrated to Vitess to solve hot spots and reduce operational complexity. Vitess's VtGate routing tier knows which table is sharded by which column — users by user_id, channels by channel_id, workspaces by workspace_id — so the PHP code no longer needs to manage routing. Multiple Vitess clusters now run in different geographic regions.
Key insight about message persistence: Every message is written to the MySQL database before it is broadcast over the WebSocket to other users. Durability comes first, delivery second. This is why Slack can recover messages even if your client disconnects mid-conversation.

Real-time messaging system

WebSocket + pub/sub architecture

Real-time delivery is the hardest part of Slack's system. Over 100 event types flow over WebSocket connections — new messages, typing indicators, file uploads, thread replies, presence changes, profile updates, and more. The system is built around a Java-based pub/sub service that distributes events across all connected clients.

Flannel: the edge caching layer

When a client first connects, it needs to download the entire workspace snapshot (every user, channel, preference, avatar URL). This became slow and expensive as workspaces grew. Slack built Flannel — an application-level caching service deployed at edge data centers worldwide. Clients connect to Flannel, which holds the workspace snapshot in memory and opens its own persistent WebSocket back to Slack's US-East core. The client gets low-latency startup, and the core servers are shielded from millions of cold-start requests.

How consistent hashing assigns channels to gateways

Each Slack channel is assigned to a specific gateway server using consistent hashing on the channel_id. This means all members of a channel connect (through Flannel) to the same gateway — making message fanout a simple in-process broadcast rather than a distributed scatter-gather. The gateway is a stateful, in-memory service that holds active WebSocket connections and pushes events the moment they arrive.

Logical clocks for message ordering

Because messages flow through a distributed system, Slack uses a vector clock (logical clock) to establish causal ordering — it can determine which message came before which even across servers. Messages are also "salted" with a unique random token to prevent duplicate delivery when a client reconnects and replays missed events.
Mobile clients skip the WebSocket connection entirely and use REST API calls with timestamps to fetch messages missed since their last session — saving battery and bandwidth.

Slack EKM
SSL termination at the message proxy level: encrypted WebSocket connections from clients are decrypted at the edge proxy, then forwarded over the internal WAN back to US-East core data center — reducing the SSL overhead on core servers. KMS key usage is logged via AWS CloudTrail for enterprise audit compliance.

Load balancing strategy

Slack uses a fleet of HAProxy instances behind a Layer 4 AWS ELB. HAProxy distributes connections to the PHP application tier. For the message service, consistent hashing (not round-robin) routes connections — the same channel always goes to the same gateway server, enabling stateful in-memory fanout.

End-to-end message flow

What actually happens when you press Enter to send a message in Slack.

1
Client sends REST POST to PHP backend
Your client sends an HTTP POST (not WebSocket) to the PHP monolith. The message content, channel ID, and workspace ID are included. The request routes through CloudFront → ELB → HAProxy → PHP app server.
HTTP POSTCloudFrontHAProxyPHP
2
PHP persists the message to MySQL (via Vitess)
Before anything else, the PHP server writes the message to MySQL through Vitess. VtGate routes the write to the correct shard based on channel_id. The message is durable on disk before delivery begins.
Vitess VtGateMySQL shardWrite-first
3
PHP notifies the Java message server
After the DB write succeeds, PHP signals the Java-based pub/sub message server that a new event is ready for the channel. Consistent hashing ensures PHP knows exactly which message server owns this channel.
Pub/subJava tierchannel_id hash
4
Message server fans out over WebSockets
The Java gateway holds open WebSocket connections for all active members of the channel. It pushes a new_msg event to every connected client simultaneously. For a channel with 8,000 members, this is 8,000 WebSocket pushes from a single in-memory broadcast.
WebSocket pushIn-memory fanout8K× per message
5
Async jobs run in the background
Meanwhile, secondary tasks are pushed to Kafka: unfurling the link preview, indexing the message in Solr for search, sending push notifications to offline mobile users, and triggering any Slack app webhooks.
KafkaKafkagateSolr indexingPush notifications
6
Recipient clients receive and render
Each recipient's client receives the WebSocket event, updates the local in-memory workspace snapshot, and re-renders the channel view with the new message. The logical clock timestamp ensures messages appear in causal order even if they arrive slightly out of sequence.
Local state updateReact re-renderVector clock ordering
Scale reality check: Slack estimated that with 4 million users each sending 50 messages/day, a single message in an 8,000-person channel triggers 8,000 WebSocket pushes. Across all workspaces, the message server handles roughly 1.6 trillion WebSocket events per day. This is why the stateful gateway + consistent hashing design is critical — without it, every message would require a distributed scatter-gather across all gateway servers.
