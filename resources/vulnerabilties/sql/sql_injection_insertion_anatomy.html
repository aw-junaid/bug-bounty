Step-by-step animation of how a malicious user inserts SQL commands into application code

1. Normal query
2. Quote escapes string
3. Command injected
4. Comment silences rest
5. UNION attack
Developer code
User data (safe)
SELECT * FROM users WHERE username = 'alice' AND password = 'mypassword'
The developer wraps user input in single quotes '. The database sees alice as pure string data and checks it against the users table. Everything works as intended.
Result: Checks if alice / mypassword matches a record. Safe, normal behaviour.
Developer code
Attacker's quote (closes string!)
SELECT * FROM users WHERE username = '' ← attacker's quote closes the string here
The attacker types a single quote ' as their username. This closes the string the developer opened. Everything after it is now outside the string — it becomes raw SQL the database will execute as code.
The string context is broken. The DB sees a syntax error — but now the attacker knows injection is possible.
Developer code
Attacker's quote
Injected SQL command
SELECT * FROM users WHERE username = '' OR '1'='1' AND password = 'anything'
Attacker types: ' OR '1'='1. Their quote closes the username string, then they append OR '1'='1' — a condition that is always true. The DB evaluates this as real logic, not data.
Result: Returns ALL users because OR '1'='1' is always true. Attacker logs in without a valid password.
Developer code
Attacker input
Injected command
Comment (kills rest of query)
SELECT * FROM users WHERE username = 'admin' -- AND password = '...' [ignored]
Attacker types: admin'--. The -- is a SQL comment character — it tells the database to ignore everything after it. The entire password check is silently removed from the query.
Result: Logs in as admin with no password at all. The password check was commented out.
Developer code
Attacker input
Injected UNION command
Comment
SELECT name, price FROM products WHERE name = '' UNION SELECT username, password FROM users -- '
In a product search box, the attacker appends a UNION SELECT that pulls from a completely different table. The original query returns nothing (empty string), but the UNION appends all usernames and passwords to the result.
Result: Page shows the entire users table — usernames, hashed passwords, emails — disguised as product search results.
